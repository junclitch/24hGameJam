# 是正報告書: AirControlAction 空中壁押し付け摩擦の検出失敗 (raycast / groundLayer 依存で見落とし)

報告日: 2026-05-07
対象: `Assets/ScrollAction/Scripts/Actions/AirControlAction.cs`
対象セッション: 2026-05-07 (ジャンプ中の壁押し付け引っかかり修正の場)

---

## 1. 事象

ユーザから「ジャンプ中に壁に接しているのに壁方向に移動しようと思うと引っかかる。直して」と依頼を受け、`AirControlAction.OnFixedTick` に壁検出を追加する修正を実施した。1回目の修正後に「直しました」と報告したところ、ユーザから「**直っていません**」と再指摘を受け、検出方式を切り替えて 2 回目を実施。

| 修正試行 | 内容 | 結果 |
|---|---|---|
| 1回目 | `Physics2D.Raycast` で `ctx.stats.groundLayer` を検出。プレイヤー中心 (`rb.position`) から `bodyCollider.bounds.extents.x + 0.05` の距離だけ入力方向にレイ | ユーザから「直っていません」 |
| 2回目 | `Rigidbody2D.GetContacts` で物理エンジンが既に解決済みの実接触を走査し、`ContactPoint2D.normal.x` の符号で「入力方向側の壁」を判定。レイヤー指定・raycast 精度に依存しない | 採用 (ユーザ検証待ち) |

1回目は**修正完了の報告に伴う play mode 検証を実施しないまま**「壁検出 raycast を入れた → 完了」で終わっていた。実機で当該症状が解消しているかをユーザの目に通す前に手放しており、ユーザの再指摘で初めて検証不足が露呈した。

---

## 2. 真の原因

### 2.1 摩擦そのものの理論は正しかったが、検出方法を「`groundLayer` 前提の raycast」で固定した

直接原因。1 回目の `Physics2D.Raycast(origin, dir, distance, ctx.stats.groundLayer)` は **壁が `groundLayer` に含まれている前提**で書いた。根拠は `WallKickAction.cs` の冒頭コメント「壁検出は PlayerStats.groundLayer を流用 (壁=地形と同レイヤ前提)」のみ。WallKick が動くこと と AirControl が同条件で動くこと は独立した検証事項なのに、隣接コードのコメントを読んだだけで「同じ手段で行ける」と自分の前提に組み込んでしまった。

実シーンで壁が groundLayer に入っているかは確認していない。仮に壁が別レイヤー (たとえば Hazard・Default など) にいれば、レイは hit を返さず、検出が常に false となり修正効果ゼロになる。

### 2.2 「最も検出漏れのない手段」を最初に選ばなかった

壁との接触は少なくとも 2 系統で検出できる:

| 種別 | 用途 | API |
|---|---|---|
| 幾何チェック | **まだ接していない**物体の検出 (壁ジャンプの発動条件、足場予測) | `Physics2D.Raycast` / `Collider2D.Cast` |
| 物理エンジン側の解決済み接触 | **既に接している**物体の状態取得 (壁押し付け抑制、滑り判定) | `Rigidbody2D.GetContacts` / `Collider2D.GetContacts` |

AirControlAction の「壁押し付け抑制」は明確に**後者の用途**で、`GetContacts` の方がレイヤー指定にも raycast 精度にも依存せず一段直接。にも関わらず、WallKick の流儀 (前者の用途で raycast) に引っ張られて誤った API を選んだ。**用途が違えば最適な API も違う** という当たり前の原則を、隣接コードのコメントが上書きしてしまった。

### 2.3 1回目の修正後に play mode 検証を通さなかった

コード変更後、`feedback_animator_e2e_verify.md`「アニメ/遷移系は単体検証だけで完了報告しない。play mode で press→release の end-to-end 観測まで通すこと」を守らず、「raycast 入れた → 完了」とだけ報告した。実機で「壁横でジャンプ → 壁方向に入力 → 落下速度が削られないか」を 1 周通してから報告すれば、検出が発火していないことに自分で気付けた可能性が高い。

### 2.4 真因仮説と修正実装の対応関係を、自分で 1 行ずつ確認していなかった

「摩擦力 = μ × 法線力。法線力は押し付け速度に比例。targetVx=0 なら法線力ゼロで摩擦ゼロ」という機構説明は頭の中では持っていたが、**(i) 仮説 (ii) 検出 (iii) アクション (iv) 副作用** の各段階を独立に詰めるべきところを、(i)(iii) の往復だけで満足してしまった。(ii) 検出が発火しなければ (iii) アクションがいくら正しくても効果ゼロ、という当たり前の依存関係を独立に検証していない。

### 2.5 既存 memory `feedback_animator_e2e_verify.md` を引かなかった

memory のタイトル「アニメ/遷移系」から、本件 (空中物理) は対象外と無意識に分類した。本来この memory の精神は「**動的に状態が遷移する系は単発検証で完了報告しない**」であり、空中→壁接触→落下 という遷移そのものに当てはまる。`20260507-000650-timer-lives-imgui-overflow.md` §5 で既に同種の反省 (memory タイトルに引きずられて適用範囲を狭めた) が記録されているのに、それを今回の場で活かせなかった。

## 4. 再発防止策

### 4.1 物理接触の検出は「物理エンジンが解決済みの接触」を第一候補にする

押し付け抑制・滑り判定など「**既に接触している状態に対する反応**」では、`Rigidbody2D.GetContacts` / `Collider2D.GetContacts` を最優先で検討する。raycast / shape cast は「**到達前に検出したい**」用途 (壁ジャンプの発動判定、足場予測など) に限定する。判断軸は「修正対象が "未接触から接触する瞬間" を捉えたいのか、"既に接触している状態" に反応したいのか」。

### 4.2 「○○レイヤ前提」を既存コメントだけで採用しない

他のスクリプトのコメント・命名から「このレイヤーには◯◯がある」と推測して layer mask を組み込むと、シーン側のレイヤー設定変更で簡単に破綻する。**レイヤーに依存しない API があるならそちらを選ぶ**のがレイヤー設定変更に強い設計。どうしても layer mask が必要な時は、コメントではなく実機 (該当シーンの該当オブジェクトの layer 値) を確認する。

### 4.3 物理修正は (i)仮説 → (ii)検出 → (iii)アクション → (iv)副作用 の 4 段で分けて検証する

- (i) 仮説: なぜその症状が出るかの機構を 1 行で説明する (例: μ × normal_impulse による vy 削り)
- (ii) 検出: その状態を判定する手段を選ぶ (例: GetContacts vs Raycast)
- (iii) アクション: 検出された時に何をするか (例: targetVx を 0 にクランプ)
- (iv) 副作用: アクションが他の挙動に与える影響 (例: 壁から離れる入力は通常通り効くか、WallKick 発動と競合しないか)

(i) と (iii) の対応だけで「修正完了」と判断すると、(ii) の取りこぼしで根治しないパターンが起きる (今回の 1 回目)。各段階を独立に「これで本当に発火するか?」と問い直す。

### 4.4 「動的に状態が遷移する系」は memory のタイトルに関わらず end-to-end 検証

`feedback_animator_e2e_verify.md` は「アニメ/遷移系」と銘打っているが、空中→壁接触→落下のように**状態遷移を含むあらゆる修正**に同じ精神が適用できる。タイトルから「自分の今回のケースは対象外」と早期判断しない。memory 参照時は「**これは今回当てはまるか?**」を必ず自問する。

### 4.5 修正後の play mode 観測を完了報告に含める

コード変更だけで「直しました」と報告しない。play mode で

- バグ症状そのもの (壁に押し付けて落下速度が削られないか) を観測
- 周辺機能 (壁から離れる入力、WallKick 発動、地上歩行) の回帰

を最低 1 周通してから報告する。MCP / 撮影で確認できない部分は「ユーザに play mode 確認を依頼」と明示する。沈黙でユーザに丸投げしない。

### 4.6 「直っていません」の再指摘は機構再走査の合図

`20260507-000650-timer-lives-imgui-overflow.md` §4.5 と同じ反省。**1 度目の修正で根治しなかった時、表面的な数値・パラメータ調整に逃げず、機構レベル (今回なら検出 API の選択) を再走査する**のを最優先にする。

## 5. ふりかえり

- 摩擦理論 (μ × normal_impulse による vy 削り) という機構レベルの仮説は **早い段階で正しく立てられていた** にも関わらず、検出側で「raycast + groundLayer」という**最も故障しやすい組み合わせ**を選んでしまった。仮説が正しくても検出が発火しなければ修正効果はゼロ、という (i)(ii)(iii) の独立性を見落とした典型ケース。
- WallKickAction が groundLayer raycast で動いている事例があるからといって、AirControlAction の用途 (= 既存接触に対する反応) に同じ手段が適切とは限らない。**用途が違えば最適な API も違う** という当たり前の原則を、隣接コードのコメントが上書きしてしまった。
- 1回目の修正後にユーザから「直っていません」と返ってきた瞬間が、**機構レベルの再走査をやり直す合図**だった。`20260507-000650` §4.5 の知見はこのケースにも普通に適用できたが、再走査を経由せずに即座に検出方式の差し替えに走れたのは、過去の是正報告書からの learnings が断片的に効いた結果と見ることもできる。
- 今回の知見「**既存接触の判定は GetContacts、未到達対象の判定は Raycast**」は今後の物理判定全般で再利用できる粒度の規律。memory 化候補。

以上。
