# 是正報告書: UnityRoom 向け WebGL Player Settings の誤適用 (Decompression Fallback)

報告日: 2026-05-06
対象: `ProjectSettings/ProjectSettings.asset` (PlayerSettings.WebGL)
対象セッション: 2026-05-06 22:30 〜 23:25 頃

---

## 1. 事象

UnityRoom 公開を前提とした WebGL Player Settings 調整の場で、`PlayerSettings.WebGL.decompressionFallback` を **false → true** に変更した。これは UnityRoom 環境では誤りで、正しくは **false** のまま。ユーザの「Decompression Fallback は false 推奨ですよ?」という指摘で発覚し、即時 false に戻した。

設定変更直後で再ビルドは未実施だったため、本番ビルドの肥大化や配信遅延などの実害は発生していない。ただしユーザの指摘がなければそのままビルド・アップロードされ、ロード時間悪化を招いていた。

---

## 2. 真の原因

### 直接原因
「サーバ側で gzip ヘッダが落ちた時の保険」という汎用 WebGL デプロイの一般論を、確認なしに UnityRoom に当てはめた。

UnityRoom の配信サーバは `.gz` ファイルに対して `Content-Encoding: gzip` を正しく付与してくる。ブラウザはネイティブで展開できる。`decompressionFallback = true` で同梱される JS デコーダは絶対に実行されないコードで、ビルドに数百KB の死荷重として乗るだけ。

### 根本原因 (なぜそうしてしまったか)

#### a. プラットフォーム固有の挙動を確認せず、一般論で結論した
UnityRoom の HTTP レスポンスヘッダ実態を確認しないまま、「保険として true が安全」という汎用アドバイスを採用した。UnityRoom は十分に情報が公開されている特定プラットフォームであり、調べれば「Content-Encoding を返す = フォールバック不要」と即座にわかった。

#### b. 「UnityRoom 推奨」と銘打った表で根拠未確認の値を提示した
ユーザに提示した表のヘッダに "UnityRoom 推奨" と書きながら、その推奨ソース (UnityRoom 公開ガイド・ユーザコミュニティの実例) を引いていない。**読み手にとっては「裏取り済み」と誤読させる強い表現**で、信頼性の偽装に近い。

#### c. 「保険」「安全側」というキーワードに引かれて思考停止した
`decompressionFallback = true` を「効果がない場面でも害もない」と思い込んだが、実際には**ビルドサイズに数百KB上乗せされる**確実なデメリットがある。「効かない保険はつけない方がいい」という基本判断ができていなかった。WebGL ビルドはローディング時間が体験を直撃するので、無効コードの上乗せは"安全側"ではなく"悪化側"。

#### d. 設定の意味を機構レベルで説明できないまま値を勧めた
`decompressionFallback` が「サーバが Content-Encoding を返せない時に JS でデコードする経路を同梱するかどうか」だと自分で言語化できていれば、「UnityRoom はサーバが返せるから不要」と即座に結論できたはず。**仕組みを説明できないものは推奨しない**を守れなかった。

## 4. 再発防止策

### 4.1 「○○推奨」と書く前にソースを引く
引けないなら「○○推奨」と書かず「一般的に言われる」「私の理解では」「未検証だが」と強さを下げる。表のヘッダに "X 推奨" と書く時は、その X の公式情報・コミュニティ実例のいずれかを参照済みであることを暗黙の前提とする。

### 4.2 設定値の機構を1行で説明できなければ提案しない
"`decompressionFallback` = サーバが Content-Encoding を返せない環境用の JS デコーダ同梱フラグ" のように、効く/効かない条件を言語化してから値を選ぶ。説明文を書こうとして詰まる場合は、自分が機構を理解していない証拠なので、推奨を出す前に確認する。

### 4.3 「保険」「安全側」「念のため」を理由にした有効化は警戒する
デフォルト false の設定を true にする時は、必ず**有効化のコスト** (ビルドサイズ・パフォーマンス・複雑性) を併記して比較する。「コストゼロの保険」はほぼ存在しないという認識で構えること。

### 4.4 対象プラットフォームが特定されている時は、汎用アドバイスを直接当てはめない
UnityRoom / itch.io / GitHub Pages / 自前サーバはそれぞれ前提が違う。前提が違えば最適値も違う。「UnityRoom」と話題が指定されたら、UnityRoom の前提 (CDN挙動、iframe 制約、canvas 仕様、推奨設定) を一度確認層として通してから、その上で汎用知識を組み合わせる順序を守る。

## 5. ふりかえり

このセッションでは、UnityRoom 固有の問題として「ボタンが反応しない / 遅い」「星位置がずれる」を扱う流れで Player Settings 全体を見直した。**個別の症状 (Run In Background = false が iframe フォーカス喪失で停止する根因) は正しく特定できた**一方、ついでに見た `decompressionFallback` を「触っといた方が安全」で雑にいじってしまった。

主たる修正以外の "ついで変更" は、依頼スコープ外の追加であり (`feedback_no_unrequested_additions.md`)、なおかつ未確認の推奨で**信頼性も低い**。今後は「ついで」で設定変更しない。確認したものだけ提案する。
