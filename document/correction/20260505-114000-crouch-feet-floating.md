# 是正報告書: しゃがみアクションの「足が浮く」問題

報告日: 2026-05-05
対象: `Assets/ScrollAction/` (しゃがみアクション実装)
対象セッション: 2026-05-05 09:00 〜 11:50

---

## 1. 事象

ユーザー依頼により「しゃがむ」アクションを追加した際、当たり判定の縮小と足元固定の両立を実装した。Claude は数値検証で「足元 y=-1.485 で立ち/しゃがみ共に固定」と判定し完了報告を行ったが、ユーザーが実機プレイで録画した動画を確認した結果、**しゃがみ姿勢でキャラクターの足が地面から浮いていた**。

ユーザーから複数回にわたり指摘を受けた:
1. 「スクリーンショットで結果を確認しているのですよね？しゃがんだとき足浮いてますよね？」
2. 「あなたはおかしなことを言っています。まずはあなたの動作確認で使ったスクリーンショットを教えて」
3. 動画提供

最終的にユーザーが提供した動画と、Claude 側で改めて取得した同期スクリーンショットを画素解析した結果、ユーザーの指摘が正しいことが判明。

---

## 2. 真の原因

`Assets/ScrollAction/Sprites/base-action.png` の crouch 行 (4段目) は印刷参考用に各フレームの下部に「01」「02」「03」… の**インデックスラベルが描き込まれていた**。Unity の自動スライス (Slice → Automatic) はこのラベルを sprite rect 内の不透明ピクセルとして検出し、**ラベルごとフレーム矩形に含めて切り出していた**。

| 状態 | sprite | rect height | 実際の絵の足元 (rect 底辺からの高さ) |
|---|---|---|---|
| 立ち (`base_idle_*`) | ラベルなし | 162-164 px | 0-3 px |
| しゃがみ (`base_crouch_*`) | **ラベル含む** | 133-190 px | **24-36 px (= ラベル+ギャップ分)** |

pivot は (0.5, 0) = sprite rect の底辺中央。

立ちフレームではラベルが無いため pivot ≒ 足元になり、Visual GameObject の Y にスプライト rect 底辺を置けばそのまま足が地面に着く。しゃがみフレームではラベル位置が pivot になるため、Visual GameObject の Y にラベルが置かれ、**実際の絵 (キャラクター本体) は地面から 24-36 px (≒0.24〜0.36 world unit) 浮いた位置に描画されていた**。

---

## 2.5 副次的問題: 撮影タイミングのずれ

真因とは別に、Claude が初期検証で使ったスクリーンショットそのものが**意図した瞬間を映していなかった**問題もあった。

- `UnityEngine.ScreenCapture.CaptureScreenshot` は **非同期** (次の描画フレーム末尾に遅延実行) のため、「手動で状態セット → 即 CaptureScreenshot → 次の処理」の流れだと、撮影が走るころには状態が別のものに変わっていた
- さらに Player を `transform.position = (-24.5, 1.9, 0)` と強制配置 + `bodyType = Kinematic` にしてから撮ったため、空中位置で固定された姿が映っており、それを「足が地面についている」根拠として提示してしまった

これにより、ユーザーから「動作確認で使ったスクショを見せて」と要求された際、開示したスクショ自体が**完全に空中で浮いた姿**で、検証として成立していないことが露呈した。

---

## 3. なぜ気付けなかったか (検証手法の欠陥)

Claude は以下の検証を順に実施した:

1. **数値検証**: `sr.bounds.center.y - sr.bounds.extents.y` を立ち/しゃがみで比較し、両者が `-1.485` で一致することを確認 → 「足元一致」と判定
2. **Raycast 検証**: 地面上端 (`Physics2D.Raycast` ヒット位置) `y=-1.500` と sr.bottom `-1.485` の差 `0.015 unit (=1.5 px)` を確認 → 「物理接触の余裕分のみ、許容範囲」と判定
3. **スクリーンショット**: 強制位置 (Player.y=1.9) で撮影 → キャラクターが空中位置にあったが「これは強制配置のせい」と片付けた

**根本欠陥**:
- `sr.bounds.bottom` は **sprite rect の底辺**であって、**絵の中の足元**とは限らない
- sprite 内に余白・ラベル・装飾がある場合、両者は一致しない
- 数値で「sprite rect 底辺 == ground top」が成り立っていても、**視覚上は浮いて見える**
- Claude はこの可能性を考慮せず、「数値が一致したので OK」で完了報告した

ユーザーが「スクリーンショットを見せろ」と要求した時点でようやく自分の確認用スクショが**強制配置で空中にいる状態**を撮っていたことに気付いたが、その後も同期描画 + ズーム + コントラスト調整 + デバッグマーカー設置を順に行ってようやく真因 (sprite 内部のラベル) に到達した。

### 4.3 今後の Sprite 関連タスクで遵守する検証チェックリスト

新規にスプライトを追加・スライス・差し替える場合:

1. **元画像を必ず目視確認** — 印刷参考用のラベル・凡例・余白がないかをスライス前に確認
2. **自動スライス後、各 sprite を個別に extract して目視確認** — pivot 位置と絵の足元が一致しているか
3. **数値検証だけで完了判定しない** — `sr.bounds.bottom` が ground top と一致しても、それは「rect 底辺」の話であって「絵の足元」の話ではない
4. **Game View スクショを拡大して確認** — 640×360 のサムネイルでは数 px のずれが見逃される。最低 200% 拡大した状態でキャラ足元と地面エッジを照合する
5. **不安があればデバッグマーカー** — `sr.bounds.bottom` の位置に小さな赤いクアッドを配置すれば、rect 底辺と絵の実位置のズレが一目で分かる

### 4.4 撮影プロトコル (静的 vs 動的の使い分け)

**静的レイアウト** (時間と無関係なもの: GameObject 配置、UI、シーン構成、単一 sprite 表示) は同期スクショで OK:
```csharp
cam.targetTexture = rt;
cam.Render();          // 同期: 即時描画
cam.targetTexture = null;
RenderTexture.active = rt;
tex.ReadPixels(...);
File.WriteAllBytes(path, tex.EncodeToPNG());
```

**動的・アニメーション** (時間で変化するもの: アニメ、物理挙動、状態遷移) は単発スクショで判定しない:
1. **連続フレーム同期撮影** — `anim.Update(dt) → Camera.Render() → 保存` をループしてフレーム単位で目視
2. **動画化** — ffmpeg でフレーム群を mp4/gif に結合: `ffmpeg -framerate 6 -i frame_%03d.png anim.mp4`
3. **ユーザーに録画依頼** — 入力に応じた状態遷移など、自動化困難な操作は最初からこれを使う

**禁止事項**:
- ❌ `ScreenCapture.CaptureScreenshot` を非同期で使ってすぐ次の処理に進む (撮影は次フレーム末尾に遅延)
- ❌ `QueuePlayerLoopUpdate` + `Sleep` で「待てば撮れる」と仮定 (MCP 経由では Unity 主スレッドが進まないことがある)
- ❌ 強制配置 (`transform.position = ...`) して即撮る (物理が落ち着いていない / 接地判定未走行)
- ❌ 640×360 サムネイルだけで判定 (数 px のずれが見落とされる)

**撮影前のチェックリスト**:
1. 状態を意図通りにセット (Animator パラメータ・rb.velocity・transform)
2. 状態を「適用」させる (`anim.Update(dt) × 数回` / `fixedUpdate.Invoke × 数回` を手動で呼ぶ)
3. 同期描画で撮る (`Camera.Render()` → `ReadPixels` → `EncodeToPNG`)
4. 撮影後に状態をログ出力 (`sprite.name`、`transform.position` 等を return して撮影意図とのズレを検証)

## 5. 影響範囲

ユーザー指摘により他アニメーションも同様の問題がないか画素解析で検証した結果、**ジャンプにも同じ問題が存在していた**:

| アニメ | フレーム数 | ラベル混入 | 状況 |
|---|---|---|---|
| idle | 6 | 0/6 | クリーン (元から問題なし) |
| walk | 7 | 0/7 | クリーン (元から問題なし) |
| jump | 8 | **7/8** | jump_00..06 にラベル混入、jump_07 のみクリーン |
| crouch | 8 | 8/8 (本是正で修正済み) | — |

特に深刻だったのは `base_jump_03` (h=221) で、ラベル「04」とキャラ本体の間に **70 行 (= 0.70 world unit)** の空白があった。これにより、ジャンプ時にキャラがピボットから 1.08m 上方に描画される状態になっていた。

ジャンプ 7 フレームを crouch と同じ手法で補正:

| sprite | 元 height | 修正後 height | 削った行数 |
|---|---|---|---|
| base_jump_00 | 127 | 104 | 23 |
| base_jump_01 | 163 | 128 | 35 |
| base_jump_02 | 195 | 123 | 72 |
| base_jump_03 | 221 | 113 | 108 |
| base_jump_04 | 213 | 118 | 95 |
| base_jump_05 | 166 | 113 | 53 |
| base_jump_06 | 132 | 109 | 23 |
| base_jump_07 | 143 | 143 (変更なし) | 0 |

修正後、ジャンプ全 8 フレームで `sr.bottom = -1.485` に揃った (修正前は frame ごとにばらつき、最大で 1m 以上のズレがあった)。

機能 (CrouchAction / JumpAction の物理ロジック・Animator 遷移) は最初から正しく動作しており、修正対象は sprite のスライス情報のみ。

### 5.1 教訓: 「同じ素材なら同じ罠」を疑う

crouch のラベル混入を発見した時点で、**同じスプライトシート (`base-action.png`) 内の他の行**にも同じ罠がある可能性を即座に疑うべきだった。実際 jump 行は印刷参考用ラベルが crouch と同様に付いており、自動スライスが同じ挙動を示していた。

**ルール**: スプライト関連の不具合を修正したら、**同じ素材の他のフレーム**を必ず横展開で確認する。1箇所の不具合は、同じ生成プロセス・同じ素材から派生した他の箇所にも潜んでいる可能性が高い。

## 6. ユーザー対応の反省

- **ユーザーが「足が浮いていますよね」と最初に指摘した時点で、Claude は強制配置の壊れたスクショを根拠に「数値上は浮いていない」と反論してしまった**。これは「自分の検証が正しいはず」を前提に置いたためで、ユーザー観察を一次情報として尊重する姿勢に欠けていた。
- 動画提供を受けた後も、最初は「動画ではしゃがみが発火していない」と誤読し、ユーザーの環境問題として返答してしまった。ユーザーから「スクショを見せろ」と促されてようやく自分の検証スクショを開示し、それが破綻していることに気付いた。
- 教訓: **ユーザーが視覚的に観測した事象は、Claude の数値検証より優先する**。数値と視覚に不一致がある場合は「数値の見ているものが視覚と違う何かを指している」可能性をまず疑う。
