# 是正報告書: Glider アニメーション用スプライト行の誤認

報告日: 2026-05-05
対象: `Assets/ScrollAction/Animations/glider.anim`
対象セッション: 2026-05-05 14:50 〜 15:05

---

## 1. 事象

ユーザーから「グライダー」アクションの追加依頼があり、`add-action.png` の **Row 2 (ムササビマント / 滑空)** のスプライトでアニメーションを組むよう指示された。

しかしこちらが選定したのは **sprite 9, 10, 11, 13, 14, 15** の 6 枚で、ユーザーから「アニメーションを見る限り、ジェットパックを使っていそう」と指摘を受けた。再調査の結果、これらは **Row 1 (Jetpack)** の左右端のフレームであり、本物の Row 2 は **sprite 84〜93** の 10 枚であった。

| | 誤選定 (Row 1 と判明) | 正しい選定 (Row 2) |
|---|---|---|
| sprite index | 9, 10, 11, 13, 14, 15 | 84, 85, 86, 87, 88, 89, 90, 91, 92, 93 |
| 枚数 | 6 | 10 |
| Y 範囲 (bottom) | 790〜799 | 537〜589 |
| Y 範囲 (top) | 906〜936 | 633〜682 |
| X 配置 | 31〜321 と 1212〜1495 (左右端のみ) | 36〜1475 (連続的) |

---

## 2. 原因分析

### 2.1 直接原因: Y 座標の解釈ミス

スプライトシートには大きな"行"が 4 段ある (Jetpack / ムササビ / Sliding / Warp)。それぞれの行は **およそ 220 px 程度の高さ**を持っており、行内のスプライトは Y bottom にバラつきがある。

| 行 | Y 範囲 (bottom) | 高さ |
|---|---|---|
| Row 1 (Jetpack) | 790〜966 | 約 220 px |
| Row 2 (ムササビ) | 537〜693 | 約 150 px |

Row 1 の sprite 1〜8 は y=832〜901、sprite 9〜15 は y=790〜799 に位置する。**bottom が約 30〜100 px ずれているだけで、同じ Row 1 に属している**。私はこの bottom のずれを「別の行」のサインだと誤認した。

### 2.2 間接原因: 状況証拠を無視した

調査中に以下の不自然な状況証拠があったが、いずれも軽視した。

1. **sprite 9〜15 の X 配置**: x=31〜321 (左端) と x=1212〜1495 (右端) に分かれ、**中央が完全に空白**。本来 1 つのアニメーションなら左→右に連続的に並ぶはずで、この空白は強い違和感だった
2. **sprite 0 (Row 1 ラベル) と sprite 1〜8 の Y top の重なり**: sprite 0 は y=966〜1016、sprite 1 は y=832〜980。ラベルとキャラクタが重なる Y 範囲を持つことから、**1 つの行は Y 方向に厚い (200px超) 構造**だと気付くべきだった
3. **sprite 84 以降の存在**: meta ファイルには sprite 84〜93 の character サイズのスプライト (w=71〜140, h=89〜115) が y=537〜682 に並んでいた。10 枚連続で X が左→右に増えており、これこそが整然とした 1 つの行であった

### 2.3 根本原因: 検証プロセスを省略した

**最も重大な不備は、選定したスプライトが本当に Row 2 かを目視確認しなかったこと**。

CLAUDE.md には次の記載がある:

> ## 撮影・確認プロトコル (完了報告前に必ず実施)
> ...
> ### Sprite 関連の追加チェック
> - 新規スプライトをスライスしたら **各フレームを個別に extract して目視** (印刷参考のラベル・ガイド線・余白が rect 内に紛れ込んでいないか)

このプロジェクトの規約として「スプライトを使うときは目視せよ」が明示されていたにもかかわらず、`add-action.png` の Read による画像表示を一度行っただけで、選定後に `extract → 目視` を行わずに完了報告した。

加えて、メモリには次の項目が記録されていた:

> [見た目だけで完了判定しない](feedback_verify_colliders.md) — スプライト/Transform 変更後は SR.bounds と BC.bounds を必ず数値検証する

この feedback の射程は colliders だが、根底にある原則は「**見た目 (今回は『meta の Y 値の数値』) だけで完了判定せず、実際に出力される結果をユーザー視点で確認する**」であった。今回はちょうどこの逆で、「数値だけ見て決めて、ユーザー視点 (= 実際にどのキャラ絵か) を確認しなかった」。

---

## 3. 再発防止策

### 3.1 即時対応 (本セッション内で実施済み)

- glider.anim を sprite 84〜93 の 10 フレームで再生成
- ループ長を 5/6 秒 (6 フレーム) → 10/6 ≒ 1.67 秒 (10 フレーム) へ修正
- m_StopTime を 1.0 → 1.6666666 に更新

### 3.2 仕組み化 (次回以降の運用ルール)

スプライトシートから一部の行/列を選ぶ作業をするときの**必須プロトコル**:

1. **画像全体の Read を行い、ユーザーが指す行/列に何が描かれているか言語化する**
   (例: 「Row 2 はムササビマントで、左から: 立ち姿 → ジャンプ予備動作 → ...」)
2. **meta の sprite 群を `(sprite_index, x, y, w, h)` の表に整理し、Y 範囲でクラスタリングする**
   - bottom Y のクラスタが見かけ上分かれても、top Y を見たり、隣接する別行の Y 範囲と比較したりして「**1 行の高さ**」を確認する
3. **選定したスプライトを 1 枚抽出して目視確認する**
   - 1 枚で十分。これが本当にユーザー指定の絵か確認できれば全件 OK
   - Unity Editor が起動できるなら `cam.Render` でテクスチャに切り出して PNG 出力
   - または add-action.png の **該当 rect (x, y, w, h)** をクロップして PNG として保存し、自分で Read で見る
4. **完了報告に「Row N に属することを確認した方法」を 1 行入れる**
   - 例: 「sprite 84 (rect x=36 y=562 w=71 h=115) を切り出して目視確認、ムササビマントの立ち姿と一致」

このうち **3 (1 枚抽出して目視確認)** は今回省略した工程であり、これを必須化することで本件の再発は防げる。

### 3.3 メモリへの記録

CLAUDE.md には sprite 目視チェックの記載があるが、**「スプライトシートから "行" を選ぶときも目視確認の対象である」**ことは明示されていなかった。本件を踏まえ、自動メモリにフィードバックとして記録する:

> スプライトシートから特定の行/列のスプライトを選ぶとき、選んだ rect が**本当にユーザー指定の見た目か** 1 枚抽出して目視確認してから完了とする。Y 座標の数値クラスタリングだけで判断しない。

---

## 4. 教訓

- **「数値だけで判断してユーザー視点で確認しない」は本プロジェクトで頻繁に踏みうる罠**。CLAUDE.md の sprite 目視チェック規約は colliders に限らず、スプライトシートの行選定にも適用すべきだった
- **不自然な状況証拠 (今回なら「中央が空白なのに端だけスプライトがある」) は仮説を疑うサインとして扱う**。「1 つの行のはずなのに不連続なのは変」と感じた時点で、別の行の可能性を検討すべきだった
- **CLAUDE.md / メモリの規約を「字面通りの対象」だけで適用すると同じ間違いをする**。"Sprite 関連の追加チェック" を colliders 用と決めつけず、原則として一般化して当てはめる癖をつける
