# グライダー (ムササビマント) アクション 実装まとめ

実装日: 2026-05-05 (14:50 〜 15:10 頃)
対象シーン: `Assets/Scenes/ScrollAction.unity`
関連: [是正報告書](../correction/20260505-150432-wrong-sprite-row-selection.md)

## 概要

ScrollAction の PlayerAction インベントリ系に「グライダー」アクションを追加した。`add-action.png` の 2 行目 (ムササビマント / 滑空) のフレームを使い、以下を実現:

- F キー長押しかつ**空中**で発動 (接地中は通常移動のまま)
- 横移動の最高速を `walkSpeed × airSpeedMultiplier` まで引き上げ (少しだけ強化)
- 落下速度を `-maxFallSpeed` で下限クランプ (緩降下)
- アニメーションは Row 2 の 10 フレームを 6fps でループ再生

実装中、スプライトシートから「Row 2」を識別する際に Y 座標 (bottom) のクラスタリングだけで判定し、実際は Row 1 の左右端フレームを誤選定した。ユーザー指摘により是正。詳細は是正報告書参照。

## 成果物

### 1. PlayerAction (新規)
`Assets/ScrollAction/Scripts/Actions/GliderAction.cs`

- `[CreateAssetMenu]` 付き ScriptableObject (HorizontalMove / Jetpack / Crouch と同じ A+B+C パターン)
- `OnFixedTick` で:
  - `ctx.gliderHeld && !ctx.isGrounded` のときだけ発動 (空中専用)
  - `Mathf.MoveTowards` で `inputX × walkSpeed × airSpeedMultiplier` をターゲットに横加速 (HorizontalMoveAction と同方式・上書き)
  - `vy < -maxFallSpeed` ならクランプ (上昇中=`vy>=0`には介入しない)
  - `ctx.isGliding = true` をセットし、PlayerController 経由で AnimatorBridge へ橋渡し
- 内部状態を持たないため `OnRespawn` / `OnSessionInit` は不要

```csharp
public override void OnFixedTick(PlayerActionContext ctx, int count)
{
    if (!ctx.gliderHeld || ctx.isGrounded) return;
    ctx.isGliding = true;

    float targetVx = ctx.inputX * ctx.stats.walkSpeed * airSpeedMultiplier;
    float newVx = Mathf.MoveTowards(
        ctx.rb.linearVelocity.x, targetVx,
        ctx.stats.acceleration * Time.fixedDeltaTime);

    float vy = ctx.rb.linearVelocity.y;
    if (vy < -maxFallSpeed) vy = -maxFallSpeed;

    ctx.rb.linearVelocity = new Vector2(newVx, vy);
}
```

## スプライト行誤認 (重要・是正済み)

### 起こったこと

`add-action.png` の Row 2 (ムササビマント) のフレームを選定する際、`png.meta` の sprite 群から bottom Y 座標が近い 6 枚 (sprite 9, 10, 11, 13, 14, 15) を選んでアニメーションを組んだ。動作確認時にユーザーから「アニメーションを見る限り、ジェットパックを使っていそう」と指摘され、実際にこれらは Row 1 (Jetpack) の左右端フレームだったことが判明した。

### 構造

| 行 | 範囲 (bottom Y) | 範囲 (top Y) | 高さ | sprite index |
|---|---|---|---|---|
| Row 1 (Jetpack) | 790〜966 | 906〜1016 | 約 220 px | 0 (label) / 1〜8 (中央) / **9〜15 (左右端)** |
| Row 2 (ムササビマント) | 537〜693 | 633〜743 | 約 150 px | 83 (label) / **84〜93** |

Row 1 は高さ 220px で、行内のフレームでも bottom Y が 30〜100px ばらつく。bottom Y のクラスタが分かれていても**同じ行に属する**ことがある (今回の sprite 1〜8 と 9〜15 がまさにこれ)。

### 見落とした状況証拠

- sprite 9〜15 の X 配置が x=31〜321 (左端) と x=1212〜1495 (右端) に分かれ、**中央が完全に空白**だった (1 つのアニメーションなら左→右に連続するはず)
- sprite 84〜93 が y=537〜682 に X=36〜1475 で**整然と 10 枚並んでいた** (こちらが本物の連続アニメーション)

### 是正

`glider.anim` の `m_PPtrCurves` と `pptrCurveMapping` を sprite 84〜93 (10 frame、internalID 6798783561900498035 〜 3381858203837025683) に書き換え、`m_StopTime` を 0.83 → 1.67 に修正。

詳細は是正報告書: `document/correction/20260505-150432-wrong-sprite-row-selection.md`。

## アーキテクチャ的な選択

### Any State → Glider を採用した理由

Crouch は接地中のみ発動するため Idle 状態からの遷移だけで足りるが、Glider は空中専用なので Idle / JumpUp / JumpDown のいずれの状態からも遷移する必要がある。各状態に Idle→Glider / JumpUp→Glider / JumpDown→Glider の 3 本を張るのは冗長で、Any State 経由なら 1 本で済む。

副作用として、Glider 終了時 (IsGliding=false) には Idle に戻る設定にしているため、まだ空中なら直後に Idle→JumpDown が発動する (Idle 状態の既存遷移が IsGrounded=false / VerticalSpeed<0.1 で発動)。50ms × 2 ≒ 100ms 程度の中間状態は許容範囲とした。

### 価格の中間設定 (40 G)

Jetpack=50 (上昇能力で強力) / Crouch=30 (基礎能力) の中間。グライダーは「ジェットパックほどではないが、フォール抑制+横移動強化」で立ち位置を取ったため。値は将来 .asset で調整可能。

## 反省・教訓

本セッションで以下の失敗をした:

1. **追加作業を user に投げた**: 初回実装で Animator State / .anim 作成を「Editor で対応してください」とユーザー作業として残した。「なぜ追加作業を与えるのか」と指摘され、自分で YAML 直編集する方針に変更
2. **スプライト行を誤選定**: meta の bottom Y クラスタリングだけで Row 2 を判定し、実際は Row 1 の左右端フレームを選んでいた (上記参照)
3. **状況証拠を軽視**: 「中央が空白で端だけスプライトがある」という違和感を「自動スライスが中央を拾えなかった」と勝手に解釈し、別行の可能性を疑わなかった
4. **CLAUDE.md の sprite 目視チェック規約を未実行**: 規約には「新規スプライトをスライスしたら各フレームを個別 extract して目視」とあったが、選定後の確認をせずに完了報告した

これらの教訓は是正報告書と memory に記録済み。
