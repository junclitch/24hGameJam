# ワープアクション 実装まとめ

実装日: 2026-05-05 (17:00 〜 18:00 頃)
対象シーン: `Assets/Scenes/ScrollAction.unity`
関連: [是正報告書](../correction/20260505-180000-warp-sprite-pivot-misalignment.md)

## 概要

ScrollAction の PlayerAction インベントリ系に「ワープ」アクションを追加した。`add-action.png` の 4 行目 (ワープ) のフレームを使い、以下を実現:

- **C キー押下フレーム**で発動 (押下フレームのみ)
- 発動中は **当たり判定 (BoxCollider2D) を完全に無効化** + **gravityScale = 0** + **velocity = 0** で落下を抑止
- `warpDuration` 秒の半ば (`teleportFraction`) で **進行方向 (`facingDir`) に `warpDistance` 先へ瞬間移動**
- 6 フレームのアニメ (前半 = 出発側で消える / 後半 = 到着側で現れる) を再生し、終了時に物理を復元
- リスポーンや起動初期化時にも安全に物理復元する仕組み (collider 無効のまま放置されない)

実装中、以下の 2 段階の問題でユーザー指摘を受けて修正した:
1. AnimatorController の fileID が **Int64.MaxValue を超過** → ID を範囲内に置き換え
2. アニメーション再生時に **キャラがスライディング同様に地面へ沈む** (pivot 不整合) → meta の alignment / pivot 修正

詳細経緯は是正報告書参照。

## 成果物

### 1. PlayerAction (新規)
`Assets/ScrollAction/Scripts/Actions/WarpAction.cs`

- `[CreateAssetMenu]` 付き ScriptableObject (Sliding / Glider / Jetpack と同じ A+B+C パターン)
- `OnFixedTick` で:
  - 非アクティブ時に `ctx.warpRequested` を見て `StartWarp(ctx)`
  - アクティブ中は `ctx.isWarping = true` を毎 Tick セット (Animator 橋渡し用)
  - `elapsed` を加算し、`elapsed >= warpDuration * teleportFraction` のタイミングで `rb.position = targetPosition` で瞬間スワップ
  - 毎 Tick `rb.linearVelocity = Vector2.zero` (gravityScale=0 と二重保険)
  - `elapsed >= warpDuration` で `EndWarp()`
- `StartWarp` で:
  - rb / collider をキャッシュ (OnRespawn 時に ctx 無しでも復元できるように)
  - `originalGravityScale` を保存し `rb.gravityScale = 0`
  - `bodyCollider.enabled = false`
  - `targetPosition = rb.position + (facingDir * warpDistance, 0)` を計算
- `OnRespawn` で `isActive` 中なら `RestorePhysics()` してからリセット (collider 無効のまま放置を防ぐ)
- `OnSessionInit` で起動時状態クリア

```csharp
public override void OnFixedTick(PlayerActionContext ctx, int count)
{
    if (!isActive)
    {
        if (ctx.warpRequested) StartWarp(ctx);
        else return;
    }

    ctx.isWarping = true;
    elapsed += Time.fixedDeltaTime;

    if (!teleported && elapsed >= warpDuration * teleportFraction)
    {
        ctx.rb.position = targetPosition;
        teleported = true;
    }

    ctx.rb.linearVelocity = Vector2.zero;

    if (elapsed >= warpDuration)
        EndWarp();
}
```

## アーキテクチャ的な選択

### teleportFraction = 0.5 で中間スワップ

「press 即瞬間移動 → 残り全部到着側演出」と「全期間 lerp 移動」の中間案。`warpDuration * teleportFraction` 経過時点で position を瞬間スワップする方式。

選んだ理由:

- 6 フレームのアニメ前半が「出発側でキャラがフェードする演出」、後半が「到着側でキャラが現れる演出」になっていたため、絵に位置スワップタイミングを合わせるのが自然
- `teleportFraction` をインスペクタで調整できるので、絵と挙動のズレが出たら数値だけで詰められる

### 物理状態を SO のキャッシュフィールドに保持

`StartWarp` 時に `activeRb`, `activeCollider`, `originalGravityScale` を `[NonSerialized]` フィールドにキャッシュし、`RestorePhysics()` でこれらを使って復元する。理由:

- `OnRespawn()` は `PlayerController.RespawnToStart()` から呼ばれ、**ctx を渡されない** API になっている (他のアクションも同様)
- ワープ中にリスポーンが走った場合、`isActive` のまま物理が無効化されたまま放置される事故を防ぐ必要がある
- ctx に依存せず復元できるよう、開始時にキャッシュしておく

```csharp
public override void OnRespawn()
{
    if (isActive) RestorePhysics();
    ResetState();
}

private void RestorePhysics()
{
    if (activeRb != null) activeRb.gravityScale = originalGravityScale;
    if (activeCollider != null) activeCollider.enabled = true;
    activeRb = null;
    activeCollider = null;
}
```

### 落下抑止に gravityScale=0 と velocity=0 の二重保険

物理的には `gravityScale = 0` だけで十分なはずだが、

- 万が一 `Update` から `rb.AddForce` 系を呼ばれても影響を残さない
- `linearVelocity` を毎 Tick 上書きすることで「ワープ中に他のアクションが速度を書き換えても無効化される」ことを保証 (HorizontalMoveAction が C キー入力時に同 Tick で vx を書き換える等のパスがあっても安全)

の 2 点から、二重に保険をかけた。

### Warp state も Any State 経由 (Sliding 同パターン)

ワープは状態を問わず発動可能 (空中・地上・歩行中・ジャンプ中 OK) なので、Any State → Warp (`IsWarping == true`) を採用。Sliding/Glider と一貫。

### 当たり判定の無効化に collider.enabled を使う

`PlayerController.SyncCollisionMask` が `bodyCollider.excludeLayers` を地面レイヤとの衝突制御に使用しているため、ワープ用の無効化に `excludeLayers` を触ると衝突。`enabled = false` で完全無効化することで影響範囲を限定した。

## 反省・教訓

本セッションで以下の失敗をした:

1. **同じ罠への再発**: スライディング実装時に `add-action.png` のキャラフレーム pivot 不整合で 1 度躓き、是正報告まで書いた直後の本セッションで再発させた。「直したのは sliding 1 枚」「同じシートの他の slice にも同じ問題が残っている」という認識が無く、warp 採用前の事前検証を怠った
2. **fileID の値域確認ミス**: 既存の sliding/glider の fileID パターンを真似たつもりが、桁数 (19 桁) は同じでも値域が Int64.MaxValue を超えていた。「桁数だけ揃える」では足りない
3. **対症療法ではなく根本治療を選べた点はよかった**: pivot 問題の発覚後、「位置 offset で逃げる」「アニメフレームを差し替える」等の対症ではなく、meta レベルでの alignment / pivot 修正という根本治療を選択。これは前回スライディング是正の経験が活きた

これらの教訓は是正報告書とメモリに記録済み。

## 次セッション以降の影響

ユーザー方針: **「今後の新規アクションは `add2-action.png` から作る」** が確定。

次セッションで「アクション追加して、add2-action.png の N 行目を使って」と依頼を受けた場合の手順:

1. `add2-action.png` 内の対象行のスライス idx を特定
2. **anim に組み込む前**に対象スライスの pivot.y(rel) を Unity API で確認
3. `pivot.y(rel) != 0` なら meta の `alignment: 0 → 7`、`pivot: {0,0} → {0.5, 0}` に修正
4. ImportAsset(ForceUpdate) 後、`bounds.min.y == 0` を再測定
5. 通常通り .anim / Player.controller / PlayerAction / .asset / Shop.prefab を編集
