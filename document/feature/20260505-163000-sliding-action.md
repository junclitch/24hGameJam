# スライディングアクション 実装まとめ

実装日: 2026-05-05 (15:10 〜 16:30 頃)
対象シーン: `Assets/Scenes/ScrollAction.unity`
関連: [是正報告書](../correction/20260505-163000-sliding-sprite-overlap.md)

## 概要

ScrollAction の PlayerAction インベントリ系に「スライディング」アクションを追加した。`add-action.png` の 3 行目 (スライディング) のフレームを使い、以下を実現:

- **Z キー押下**で接地中のみ発動
- 当たり判定 (BoxCollider2D 高さ) を縦に縮小
- 横方向の慣性を活かして滑走、地面摩擦で線形減速
- **横方向のスピードが大きいほど初速・距離が伸びる**
- 静止状態でも staticBoostSpeed の固定初速で 1秒滑り出せる
- アニメーションは `add-action_156` 1枚を固定表示 (滑走中は同じ絵)

実装中、複数の問題でユーザーから指摘を受けて段階的に修正した:
1. 初回フレーム選定 (9枚ループ) で立ち姿勢が混じってちぐはぐ → 低姿勢ループ → 1枚固定 へと変更
2. スプライト pivot 不整合で地面にめり込み → pivot 修正
3. HorizontalMoveAction との二重減速で 0.3 秒で停止 → 線形減速の絶対計算に変更
4. 走行スピードが距離に反映されない → speedMultiplier 上げ + maxDuration 延長
5. Z+方向キーの初速ブースト弱小 → 初速を `Max(慣性, staticBoost)` 化

詳細経緯は是正報告書参照。

## 成果物

### 1. PlayerAction (新規)
`Assets/ScrollAction/Scripts/Actions/SlidingAction.cs`

- `[CreateAssetMenu]` 付き ScriptableObject (Crouch / Glider と同じ A+B+C パターン)
- `OnFixedTick` で:
  - `ctx.slidingRequested && ctx.isGrounded` で開始判定
  - **線形減速の絶対計算**: `currentSpeed = startSpeed - deceleration × elapsed` を毎 Tick 計算し `rb.linearVelocity` を上書き
  - 終了条件: 離地 / 時間切れ / 速度下限割れ
  - `ctx.isSliding = true` をセットし、PlayerController 経由で AnimatorBridge へ橋渡し
- `StartSliding` で初速・方向決定 + コライダー縮小 + タイマーリセット

```csharp
private void StartSliding(PlayerActionContext ctx, BoxCollider2D box)
{
    float vx = ctx.rb.linearVelocity.x;

    if (Mathf.Abs(ctx.inputX) > 0.01f)
        slideDir = Mathf.Sign(ctx.inputX);
    else if (Mathf.Abs(vx) > 0.01f)
        slideDir = Mathf.Sign(vx);
    else
        slideDir = ctx.facingDir;

    startSpeed = Mathf.Max(Mathf.Abs(vx) * speedMultiplier, staticBoostSpeed);
    ctx.rb.linearVelocity = new Vector2(slideDir * startSpeed, ctx.rb.linearVelocity.y);

    ApplyShrunkCollider(box);
    elapsed = 0f;
    isActive = true;
    ctx.isSliding = true;
}
```

コライダー縮小は CrouchAction と同じ「底面 (足元) を保ったまま縦縮小」方式。

## アーキテクチャ的な選択

### 線形減速の絶対計算 (HorizontalMove との二重減速回避)

最初の実装は `Mathf.MoveTowards(vx, 0, deceleration*dt)` で減速していた。しかしこれだと HorizontalMoveAction の `Mathf.MoveTowards(vx, 0, acceleration*dt)` と同じ Tick 内で**二重に**減速され、設計上 1秒滑るはずが 0.3秒で停止していた。

| アクション | 1Tick の減速 | 設定値 |
|---|---|---|
| HorizontalMoveAction | acceleration × dt = 1.6 unit/sec | acceleration = 80 |
| SlidingAction (旧) | deceleration × dt = 0.16 unit/sec | deceleration = 8 |
| 合計 | 1.76 unit/sec/Tick (実効減速 88 unit/sec²) | |

是正として **絶対計算方式**に変更:

```csharp
elapsed += Time.fixedDeltaTime;
float currentSpeed = startSpeed - deceleration * elapsed;  // 線形減速の絶対値
ctx.rb.linearVelocity = new Vector2(slideDir * currentSpeed, ctx.rb.linearVelocity.y);
```

HorizontalMoveAction が `vx` を変更しても、SlidingAction が次 Tick で `currentSpeed` を再計算して上書きするため、影響を受けない。`MoveTowards` だと「現在値からの相対変化」になるが、絶対計算なら「経過時間からの一意な値」になる。

### 初速を `Max(慣性ブースト, staticBoost)` に統一

立ち止まりから「方向キー + Z」を同フレームで押されると、HorizontalMoveAction が 1Tick で `vx` を 1〜2 unit/sec まで加速 → SlidingAction が「移動中」と誤判定して `vx × 1.5` ≒ 2〜3 の弱い初速で滑り出す問題があった。

是正として、初速計算を以下に変更:

```csharp
startSpeed = Mathf.Max(Mathf.Abs(vx) * speedMultiplier, staticBoostSpeed);
```

これで:
- 静止 (vx=0) → max(0, 10) = 10 (staticBoost)
- 低速加速中 (vx=2) → max(4, 10) = 10 (staticBoost)
- 走行中 (vx=7) → max(14, 10) = 14 (慣性ブースト)

低速時には常に staticBoost が、十分な走行スピードがあれば慣性ブーストが効く。

### 方向決定の優先順

`StartSliding` で **入力 → 慣性 → facingDir** の優先順:

```csharp
if (Mathf.Abs(ctx.inputX) > 0.01f) slideDir = Mathf.Sign(ctx.inputX);
else if (Mathf.Abs(vx) > 0.01f)    slideDir = Mathf.Sign(vx);
else                               slideDir = ctx.facingDir;
```

- Z+方向キー: 入力方向に滑る (HorizontalMove で vx が中途半端でも入力優先)
- Z単押し (慣性で動いてる): 慣性方向に滑る (急ブレーキ中も慣性優先)
- Z単押し (完全静止): facingDir 方向に滑る

### Sliding state も Any State 経由 (Glider 同パターン)

スライディングは接地中のみ発動するので Idle 経由でも足りるが、Glider と整合性を取るため Any State → Sliding (`IsSliding == true`) を採用。Walking 中・JumpDown 直後など複数状態からの遷移を 1本で表現できる。

### スプライトを 1枚固定にした理由

ユーザー要望「滑走中は絵を固定」が直接の理由だが、副次効果として:

- 複数フレームを使うと、各フレームの pivot 整合チェックが増える (本実装で `_156` のみ pivot を修正済みだが `_154`, `_155` は未修正のため使えない)
- 滑走中はキャラのポーズに変化が無くて自然 (土煙だけ動かしたい場合は将来の課題)

## 反省・教訓

本セッションで以下の失敗をした:

1. **pivot.y(rel) の整合性チェック省略**: 新スプライトシート `add-action.png` を採用するとき、既存 `action.png` (idle 系) と pivot 設定が違う可能性を疑わなかった。Sprite.pivot の絶対値 (中心座標) は手元にあったが、idle 系と比較せず「中央 pivot は自然」と片付けた
2. **過去の罠 (しゃがみ実装の pivot 問題) の一般化失敗**: メモリに「pivot は常に検証対象」の記録があったが、「sprite 内のラベル問題」と狭く解釈し再発させた
3. **アクション間相互作用の未検証**: SlidingAction を単体テストでは正しく 1秒滑る計算だったが、HorizontalMoveAction と同じ Tick 内で衝突することを考慮せず、実機で 0.3秒に短縮した
4. **対症療法と根本治療の混同**: 「めり込み」報告 1回目で立ちポーズフレームを除いたのは表面の対処で、真因の pivot 不整合は解消されていなかった。2回目の指摘でようやく根本に到達した

これらの教訓は是正報告書とメモリに記録済み。
